# Dify / n8n / Claude SDK / Codex 怎么选

先不要问“哪个工具更强”。

真正应该问的是：

**你的工作流里，哪一段需要 AI 判断，哪一段需要系统稳定执行。**

## 简单判断

| 需求 | 更适合 |
| --- | --- |
| 做一个 AI 问答应用 | Dify |
| 做跨系统自动化 | n8n |
| 做深度工具调用和 Agent 执行 | Claude SDK |
| 处理本地项目、文件、代码、文档 | Codex |
| 做有状态、多节点、可恢复编排 | LangGraph |

## Dify 适合什么

Dify 适合把一个 AI 能力包装成应用。

典型场景：

- 知识库问答
- 客服问答
- 文档总结
- 简单审批前置判断
- AI 问数原型
- 内容生成应用

它的优势是上手快，界面友好，适合把 prompt、知识库、模型和 workflow 拼起来。

它的边界是复杂工程控制能力有限。比如强权限、复杂状态、细粒度日志、复杂测试回归，通常还需要外部系统配合。

## n8n 适合什么

n8n 更像自动化胶水。

典型场景：

- 每天定时抓取数据
- 收到表单后自动分类
- 生成结果后发飞书 / 邮件 / 企业微信
- 更新 Notion / 飞书表格 / CRM
- 把多个系统串成一条线

它的优势是集成能力强，适合“触发器 + 条件判断 + 系统动作”。

它不适合承载复杂推理本身。AI 可以作为其中一个节点，但不应该让 AI 控制所有外部动作。

## Claude SDK 适合什么

Claude SDK 适合做更像“工程任务执行器”的 Agent。

典型场景：

- 读取项目文件
- 调用工具
- 修改代码
- 生成脚本
- 按步骤处理复杂任务
- 对输出做多轮修正

它的优势是灵活，适合深度自动化。

但越灵活，越需要边界：工具权限、执行日志、失败重试、人工确认，都要设计清楚。

## Codex 适合什么

Codex 适合处理本地项目和工程化任务。

典型场景：

- 修改代码
- 分析 CSV
- 生成文档
- 整理知识库内容
- 扫描文件夹
- 自动化本地流程

它不是一个“网页工作流平台”，更像一个能读写本地项目的工程助手。

## 一个实际组合

比如“用户加微信要资料”这个场景：

```text
用户消息
  -> n8n 触发
  -> AI 判断用户意图
  -> 规则判断是否发试读
  -> 飞书/微信记录线索
  -> 生成个性化回复
  -> 人工确认后发送
  -> 后续跟进提醒
```

这里：

- n8n 负责流程和系统集成
- AI 负责理解和回复生成
- 规则负责控制话术边界
- 飞书负责线索沉淀
- 人负责关键发送确认

这才是 workflow，而不是打开聊天框重新问一遍。
