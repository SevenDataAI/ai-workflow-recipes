# AI 生成 SQL 审查工作流

## 场景

AI 问数和 AI 取数都会生成 SQL。

但 SQL 能跑不代表能上线。

常见风险：

- 缺分区导致全表扫描
- GMV 口径用错字段
- 商品行粒度重复计算订单数
- 查询手机号、地址、身份证等敏感字段
- join 维表导致金额放大
- 没有限制结果行数
- 使用已废弃表

这个 workflow 的目标是：AI 生成 SQL 后，先过规则门禁，再允许执行或人工审核。

## Workflow

```text
用户问题
  -> AI 生成 SQL
  -> SQL 解析
  -> 表和字段检查
  -> 分区检查
  -> 指标口径检查
  -> 权限和敏感字段检查
  -> 成本风险检查
  -> 允许执行 / 拒绝 / 人工审核
```

## AI 适合做什么

- 理解用户问题
- 生成 SQL 草稿
- 解释 SQL 意图
- 对 Bad Case 做归因

## 规则必须做什么

- 禁止敏感字段
- 必须带分区
- 大表必须限制范围
- 指标必须匹配口径库
- DWD 明细表订单数必须去重
- 危险 DDL / DML 直接拦截

## 输出

审查结果建议包含：

- `allow`: 是否允许执行
- `risk_level`: 风险级别
- `blocked_rules`: 命中的阻断规则
- `warnings`: 警告项
- `safe_sql`: 修正建议
- `review_reason`: 给人的解释

## 适合放到什么工具

- Dify：AI 问数链路里的 SQL Review 节点
- n8n：审批和通知流程
- Claude SDK / Codex：复杂 SQL 审查和自动修正
- 数据平台：真正执行前的网关
